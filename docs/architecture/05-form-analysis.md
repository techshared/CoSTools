# 四模块 × 七形态的评分与理由

> 形态池：桌面 Web / 移动 Web / 飞书 Bot / 飞书小程序 / 微信小程序 / 移动 App / TV 大屏
>
> 评分规则：0-10 分，基于交互复杂度、使用频率、使用场景、数据敏感性、协作属性五维评估
>
> 移动端判断依据详见: [06-mobile-analysis.md](./06-mobile-analysis.md)

> **2026年更新**: 飞书小程序已被飞书平台标记为"不推荐"（deprecated），实际实现改用**飞书 Web App（iframe 内嵌）**替代。评分逻辑不变，小程序相关的"7-8分"在实现中对应 Web App。

---

## 模块一：收益分成计算引擎

> 核心活动：项目创建 → 利润录入 → 成本确认 → 权重配置 → 分成计算 → 审批 → 全员查看

| 形态 | 评分 | 选择/不选择 | 核心理由 |
|------|:----:|:-----------:|---------|
| **桌面 Web** | **9/10** | **✅ 主形态** | 操作链路长（6+步），需要大屏反复核对数据；管理员配置权重、查看明细、审批确认，Web 的表格/表单/搜索/筛选体验远优于其他形态。 |
| 移动 Web | 3/10 | ❌ 不选择 | 管理端操作（创建项目/录入回款/审批）字段多、金额敏感、需反复核对，手机表单体验极差且易出错；查看端已被飞书小程序覆盖。详见 mobile-analysis#模块一。 |
| 飞书 Bot | 3/10 | ❌ 不选择 | 一条命令无法完成分成配置，需要多轮对话式输入，用户体验割裂；金额数据在 IM 中展示安全风险高。 |
| **飞书小程序** | **7/10** | **✅ 辅助形态（查看专用）** | 成员不需要配置操作，只需查看自己的分成明细；飞书工作台入口方便，"收到通知 → 打开查看"路径短；但小程序不适合做数据录入和审批。 |
| 微信小程序 | 1/10 | ❌ 不选择 | 财务数据敏感，不能在微信中流转；工作场景不应外溢到个人社交工具。 |
| 移动 App | 2/10 | ❌ 不选择 | 交互复杂度高，小屏无法承载；开发维护成本高，收益低；飞书小程序已覆盖移动查看需求。 |
| TV 大屏 | 2/10 | ❌ 不选择 | 涉及个人收入明细，不适合公开展示；分成是低频操作，不需要实时大屏。 |

### 用户旅程

```
管理员（桌面 Web）:
  打开后台 → 创建项目 → 录入回款 → 确认成本 → 调整权重 → 触发计算 → 审批 → 全员收到通知

成员（飞书小程序）:
  飞书通知 → 点击进入小程序 → 查看"我的分成" → 查看明细 → 确认/申诉
```

## 模块二：异步决策记录器

> 核心活动：决策创建（快速/完整）→ 讨论 → 投票 → 决策 → 搜索回顾

| 形态 | 评分 | 选择/不选择 | 核心理由 |
|------|:----:|:-----------:|---------|
| **桌面 Web** | **8/10** | **✅ 主形态（浏览/搜索专用）** | 查看决策历史、全文搜索、时间线浏览、关联图谱——这些深度操作需要大屏；Web 的分页/筛选/标签管理体验完整。 |
| 移动 Web | 3/10 | ❌ 不选择 | 创建决策被飞书 Bot 覆盖（一句话搞定），查询决策也是 Bot 更快（@搜索），Web 仅做深度浏览回顾——天然在桌面大屏。移动 Web 会被 Bot 和消息卡片"夹击"，做了也没人用。详见 mobile-analysis#模块二。 |
| **飞书 Bot** | **9/10** | **✅ 主形态（创建/查询专用）** | **核心创新点。** 决策发生在群聊讨论中，Bot 是最短路径：发一条消息就能记录，Bot 自动结构化。查询也直接 @Bot "搜索关于XX的决策"，无需离开 IM。 |
| 飞书小程序 | 6/10 | ⚡ 备选（轻量浏览） | 可在飞书工作台浏览决策列表，但既不如 Bot 快捷，又不如 Web 完整；属于"有它更好，没它也行"。 |
| 微信小程序 | 2/10 | ❌ 不选择 | 决策讨论不在微信发生，在微信中查看决策日志的使用场景不存在。 |
| 移动 App | 1/10 | ❌ 不选择 | Bot + Web + 小程序三者已覆盖所有使用场景，App 是纯成本。 |
| TV 大屏 | 3/10 | ❌ 不选择 | 决策时间线做成了"组织历史墙"可能有文化价值，但 ROI 太低，不是必需品。 |

### 用户旅程

```
飞书 Bot 路径（日常高频 - 手机/桌面均可）:
  群聊讨论 → @决策机器人 "记录决策：..." → Bot 确认并结构化 → 存入数据库 → 自动通知相关人

Web 路径（深度使用 - 桌面）:
  打开决策中心 → 搜索关键词 → 查看历史决策 → 关联图谱 → 点击查看详情
```

## 模块三：自驱力仪表盘

> 核心活动：每日看一眼团队健康度 → 下钻详情 → 查看趋势 → 周报洞察

| 形态 | 评分 | 选择/不选择 | 核心理由 |
|------|:----:|:-----------:|---------|
| **桌面 Web** | **9/10** | **✅ 主形态** | 仪表盘的核心是图表展示，需要大屏空间；多维度下钻交互（点击→筛选→展开）在 Web 端体验最成熟。 |
| 移动 Web | 4/10 | ❌ 不选择 | "每日扫一眼"手机场景已被飞书小程序覆盖，小程序在移动端的体验（原生组件、离线缓存、流畅加载）远优于响应式 Web；凡是需要"下钻/对比/分析"的深度操作，用户会回到电脑前。何必让 Web 同时做桌面和移动？详见 mobile-analysis#模块三。 |
| 飞书 Bot | 4/10 | ❌ 不选择 | 可推送周度健康摘要和异常预警，但无法展示图表；作为"通知渠道"有价值，但不能作为"查看渠道"。 |
| **飞书小程序** | **8/10** | **✅ 辅助形态（日活入口）** | 飞书是日常 IM，工作台小程序是每日必经入口；展示 5 个核心指标卡片（OKR 完成率/任务交付/文档活跃/协作密度/代码健康），点击进入 Web 看详情。 |
| 微信小程序 | 1/10 | ❌ 不选择 | 工作数据不应在个人微信中展示，这是原则问题。 |
| 移动 App | 3/10 | ❌ 不选择 | 响应式 Web + 飞书小程序已覆盖移动场景，独立 App 增加 3 倍开发成本。 |
| **TV 大屏** | **7/10** | **✅ 可选（文化加分）** | 办公室挂一个电视，展示团队健康度实时数据；提升透明感和归属感，是不打卡文化的重要视觉符号；但不是必须，Phase 2 再考虑。 |

### 用户旅程

```
每日路径（飞书小程序 - 手机）:
  打开飞书 → 工作台 → 点仪表盘 → 扫一眼5个核心指标 → 没问题关闭

深度路径（桌面 Web）:
  进入仪表盘 → 查看某指标趋势 → 下钻到团队详情 → 查看周报洞察

管理者路径（桌面 Web）:
  发现协作密度下降 → 点开协作网络 → 识别孤岛节点 → 飞书发起1对1沟通
```

## 模块四：成员贡献图谱

> 核心活动：查看个人贡献趋势 / 浏览团队协作网络 / 月度复盘使用

| 形态 | 评分 | 选择/不选择 | 核心理由 |
|------|:----:|:-----------:|---------|
| **桌面 Web** | **9/10** | **✅ 主形态** | 图谱可视化（D3.js/vis-network）在 Web 端生态最成熟，需要大屏进行缩放/拖拽/筛选等交互操作；贡献多维展开（按项目/按维度/按时间）需要 Web 的布局灵活性。 |
| 移动 Web | 2/10 | ❌ 不选择 | 图谱可视化是天然的大屏场景，手机屏幕物理限制决定了无法承载（不是代码能解决的）。轻量查看需求（个人贡献分）已被飞书消息推送 + 小程序覆盖。手机上勉强展示缩略图，交互体验极差，反而损害产品口碑。详见 mobile-analysis#模块四。 |
| 飞书 Bot | 2/10 | ❌ 不选择 | 无法展示图谱，文本形式的贡献分数信息密度极低。 |
| **飞书小程序** | **6/10** | **✅ 辅助形态（轻量展示）** | 不能展示图谱交互，但可以展示个人贡献简版卡片："本周贡献分 85，环比 +5%，其中代码占比 40%"，适合每周推送通知后的快速查看。 |
| 微信小程序 | 1/10 | ❌ 不选择 | 同上，工作数据不应在微信。 |
| 移动 App | 2/10 | ❌ 不选择 | 图谱在小屏上交互体验极差，独立 App 开发成本高。 |
| TV 大屏 | 4/10 | ❌ 不选择 | 可展示团队协作网络动图（文化展示），但贡献分数涉及个人评价，不适合公开显示。 |

### 用户旅程

```
个人路径（手机 → 桌面）:
  收到周贡献报告 → 点击进入飞书小程序 → 查看个人贡献概览
  → 点击"查看详情"跳转 Web → 查看完整图谱和时间线

管理者路径（桌面 Web）:
  打开贡献图谱 → 按项目筛选 → 查看协作网络 → 识别高贡献者和孤立节点
  → 点击成员查看其贡献维度分布 → 用于分成参考和 1on1 沟通

月度复盘（桌面 Web）:
  打开贡献趋势 → 选择本月范围 → 查看团队贡献热力图
  → 对比各维度投入变化 → 结合分成数据复盘
```

## 总体架构原则

```
                 ┌──────────────┐
                 │   桌面 Web    │ ← 深度操作
                 │ (4/4模块主形态)│
                 └──────┬───────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   ┌────▼────┐   ┌─────▼─────┐   ┌─────▼─────┐
   │ 飞书Bot  │   │飞书小程序  │   │ 消息卡片   │
   │ 快捷录入  │   │ 移动查看   │   │ 通知推送   │
   │ 决策记录器 │   │ 4/4模块   │   │ 扫一眼即走 │
   └─────────┘   └───────────┘   └───────────┘

   手机端三件套，覆盖所有移动场景，不做移动 Web
```

## 多形态兼容策略

同一模块的多个高分形态**不是互斥选项，而是互补拼图**。

### 兼容矩阵

| 模块 | 兼容的高分形态 | 角色分工 | 兼容度 |
|------|--------------|---------|:------:|
| **1. 收益分成引擎** | 桌面Web(9) + 飞书小程序(7) | Web=管理员操作 / 小程序=成员查看 | **完美兼容**——角色天然隔离 |
| **2. 异步决策记录器** | 飞书Bot(9) + 桌面Web(8) + 小程序(6) | Bot=快速创建查询 / Web=深度浏览 / 小程序=备选查看 | **完美兼容**——三段式深度分层 |
| **3. 自驱力仪表盘** | 桌面Web(9) + 飞书小程序(8) + TV(7) | Web=深度分析 / 小程序=每日速览 / TV=文化展示 | **完美兼容**——场景互补 |
| **4. 成员贡献图谱** | 桌面Web(9) + 飞书小程序(6) | Web=全量图谱 / 小程序=轻量分数 | **可兼容**——小程序价值有限，Phase2考虑 |

### 兼容原则：同一后端，不同前端

```
所有形态共享同一套 API + 数据库
          │
    ┌─────┼─────┐────┐
    │     │     │    │
 桌面Web Bot  小程序 TV

用户在不同场景走不同入口，数据互通、体验一致。
```

## 代码共享策略

### 不需要独立的三套代码

不同形态的代码不是完全独立的，也不是完全共享的。正确做法是**分层共享**：

```
桌面 Web ←──── 共享类型/API/工具 ────→ 飞书小程序
     │                                      │
     └───────── 共享类型/API ───────────────→ 飞书 Bot
```

### 各形态代码复用率

| 形态 | 复用代码 | 独立代码 | 说明 |
|------|---------|---------|------|
| 桌面 Web | — | ~100% | 体系最完整，独立一套 Next.js |
| 飞书小程序 | 类型+API+工具 ≈ 30% | ~70% | 不同 UI 框架（飞书 DSL），共享数据和业务层 |
| 飞书 Bot | 类型+API ≈ 20% | ~80% | 无 UI，纯服务端逻辑 |
| TV 大屏 | 与桌面 Web 共享 ≈ 90% | ~10% | 在 Web 上加"全屏展示模式"即可 |

### 不做什么

```
❌ 不要试图做 "Write once, run everywhere" 的跨端框架
   - React Native / Taro / uni-app 都不适合
   - 飞书小程序必须用飞书 DSL，无法跨端
   - Web 的 D3 图谱无法在小程序上渲染

✅ 只在 monorepo 里把类型/API/工具函数抽出来共享
   这样修改 API 返回字段时，TypeScript 能帮你检查所有端是否都改对
```

### 工程投入估算

```
桌面 Web    → 1 个前端 (全栈)     → 4-6 周 MVP
飞书小程序  → 0.5 个前端 (兼职)   → 2-3 周（在 Web 后端完成后开始）
飞书 Bot    → 0.5 个后端 (兼职)   → 1-2 周
TV 大屏     → Web 已有，加个 CSS 模式 → 1 天

总共：桌面 Web 完成后，多端复用 ≈ 增加 30% 开发量
```

## 工程实施建议

```
Phase 1 (MVP, 4-6周):
  桌面 Web: 4 个模块全部上线（不做任何移动端适配）
  数据验证：确认模式跑通，用户真的在用
  不要碰任何移动端

Phase 2 (增强, +2-3周):
  飞书 Bot: 决策记录器 Bot 上线
  飞书小程序: 分成查看 + 仪表盘速览 + 贡献轻量展示
  开始抽取 shared-types / api-client / utils 共享包

Phase 3 (完善, 按需):
  TV 大屏: 仪表盘办公室展示
  更多数据源: Figma/Notion/Linear 连接器
  AI 洞察: 自动生成周度团队健康报告

  ⚠️ 在做完所有上述功能之前，绝对不要碰移动 Web
```
